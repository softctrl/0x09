O desenvolvimento de um sistema mobile vai além do que simplesmente entregar uma solução aceitável para os seus usuários. Não desenvolvemos algo de que se precisa, mas sim criamos uma nova necessidade que impulsione o nosso público-alvo a utilizar o novo sistema de modo a não querer fazer a mesma atividade da forma antiga.
Devemos entender não só como funciona o mundo mobile, mas também como funciona a mente do usuário mobile. Assim como todos os outros usuários de sistemas, sejam desktop ou web, os usuários pouco se importam a respeito da tecnologia por trás dos programas. O que eles querem acima de tudo é que a interface seja bonita e amigável, que o sistema seja suficientemente rápido para atender suas requisições, que seja extremamente intuitivo e em último caso, mas não menos importante, que seja seguro.
Quando se decide por criar uma solução mobile o que se deseja também é passar uma imagem de empresa moderna e que esta sintonizada com o mercado atual, que busca facilitar as rotinas dos seus usuários e se manter de certa forma mais presente no dia a dia do mesmo. Ao dar este passo tomando as decisões de maneira desinformada, podemos obter um resultado consideravelmente custoso além de culminar em usuários insatisfeitos tomados a uma experiência e performances precárias.
Muito tem se falado a respeito das plataformas Android/iOS/Windows Phone e outros menos conhecidos. Neste turbilhão de plataformas, linguagens de programação e paradigmas de desenvolvimento, surgem propostas que supostamente trariam benefícios em detrimento de outros.
É válido levar em conta toda a extensão do que se deseja desenvolver para com base nestes dados poder pesar os prós e contras de cada solução. Neste ponto deve-se levar em consideração três pontos antes de decidir que conjunto de tecnologias/paradigmas e linguagens se deve utilizar. Após realizar o levantamento do que é preciso desenvolver, somado aos níveis de segurança e desempenho juntamente com a parte visual, é que se pode decidir qual tecnologia se comporta melhor na resolução do meu problema.
Antes de identificar estes pontos se faz necessário listar quais os casos de uso precisam ser desenvolvidos para a solução proposta. A partir destes dados podemos evidenciar quais as reais necessidades do sistema proposto.
Os pontos expostos são:
1) Segurança: Neste quesito é visível a diferença entre as aplicações web e as nativas. A cada dia, as regra de negócios vem sendo expostas devido a erros de desenvolvedores, mesmo que de forma inadvertida. Isto deixa brechas que podem ser usadas por pessoas com algum conhecimento técnico na tentativa de causar dano no sistema ou mesmo ao usuário. Aplicativos nativos são mais difíceis de adulterar, pois possuem uma gama completa de tecnologias desenvolvidas para proteger dados sensíveis. Neste ponto o HTML5 peca pelo fato de não possuir um mecanismo efetivo para proteção de dados em cache. O que implica que em casos de comprometimento físico de aparelho, os dados sensíveis da aplicação estão todos expostos.
2) Performance: Aqui temos uma questão de o quanto queremos que a aplicação seja utilizada. Afinal, porque pagar caro pelo desenvolvimento de um produto se não existe o interesse da utilização massiva do mesmo por parte dos potenciais usuários? Uma aplicação web desktop (cujo acesso se dá por meio de um browser desktop) que não corresponde aos padrões de velocidade esperados pelo usuário se torna tolerável pela sensação de que não existe meio melhor de realizar tal tarefa. Já uma aplicação mobile que não atenda a exigência do usuário se torna descartável pois sendo lento existe opção melhor: (a aplicação web desktop).
3) Consumo Banda/Bateria: Os dispositivos mobile atualmente possuem tanto recurso de processamento que é difícil encontrar um modelo cuja bateria dure mais de 2 ou 3 dias. Outro ponto importante é que poucos são os usuários que possuem conexão 3G, e mesmo os que possuem também possuem limitação de consumo de banda relativo ao contrato assinado com a operadora da linha celular utilizada. Mesmo com o vislumbre do 4G, ainda não será uma realidade para a grande maioria dos usuários de smartphones no Brasil devido ao preço alto. Mesmo utilizando certas estratégias, a situação sempre será como a que temos acima: mais bytes terão de ser transferidos para poder apresentar a mesma informação, por que numa aplicação web sempre estamos enviando a "tela" juntamente com os dados, enquanto que numa aplicação nativa somente os dados são entregues, bastando apenas “desenhar” a tela para o nosso usuário.
Com base em experiências próprias e de outras empresas, podemos montar a tabela a seguir onde listamos uma serie de prós e contras em alguns grupos importantes:
NATIVO
ATUALIZACOES
Prós
 Logística complicada
Sistemas nativos devem passar por um rigoroso esquema de controle de atualizações devido ao fato de que o modo de operação exige iteração com o usuário para que o processo de atualização ocorra. Isso gera um controle maior na qualidade do sistema entregue ao seu usuário.
Contras
 Logística complicada
Sistemas nativos devem passar por um rigoroso esquema de controle de atualizações devido ao fato de que o modo de operação exige iteração com o usuário para que o processo de atualização ocorra. Isto aumenta o prazo para entrega do sistema e das futuras versões aos usuários.
DISTRIBUICAO
Prós
 Formas importantes de distribuição
As aplicações nativas possuem como forma de distribuição os repositórios oficiais de suas marcas, o que garante visibilidade superior em detrimento de sites de terceiros.
PERFORMANCE
Distribuição do processamento
visual
Visto que app mobile de modo algum deve possuir regras de negócios embutidas, o aparelho se responsabilizará em processar a visualização dos dados solicitados pelo usuário. Neste ponto será possível utilizar todo o potencial de processamento dos modernos smartphones e além de ganhar tempo, se consome menos a banda de dados dos usuários, pois assim menos dados será trafegado entre o servidor e o dispositivo móvel.
SEGURANCA
 Dados trafegados
Inúmeras técnicas de proteção podem e devem ser utilizadas para proteger a confidencialidade dos dados trafegados entre o servidor e o cliente nativo que está instalado no smartphone.
VISUAL
 Interfaces ricas
Apps nativas podem utilizar de todo o potencial visual oferecido pelos smartphones atuais que possuem um apelo visual muito grande. Todo o expertise em IHC está disponível para simples uso nas aplicações nativas.
WEB
ATUALIZAÇÕES
Prós
 Simplicidade nas atualizações
Aplicações web tem o poder de garantir que a qualquer momento que o seu usuário acessar o programa estará utilizando a versão mais atual em produção.
CROSS
Contras
 Suporte inconsistente de browser
Sendo uma app 100% web, o meio de utilização se daria pelo browser nativo dos dispositivos móveis. Neste caso nem todos dão suporte idêntico a todas as tecnologias web, o que no final iria gerar um pequeno mas importante controle a respeito dos diversos dispositivos a serem utilizados.
Prós
 Universalidade entre plataformas
Todo dispositivo móvel possui suporte ao markup html, o que permite que o sistema seja desenvolvido somente uma vez e que rode em diversos smartphones no mercado.
CUSTO
Redução de custo para projetos
simples
Em projetos simples (considere como sendo projetos unicamente web utilizando css responsivo) podemos obter uma redução do custo de desenvolvimento bem como de manutenção.
DISTRIBUIÇÃO
Contras
 Distribuição penalizada
As lojas de software mantidas pelas diversas plataformas mobile somente aceitam a distribuição de aplicações nativas, o que obriga ao dono de uma app web ter de criar os seus próprios meios de disseminação e marketing para tentar conquistar usuários em potencial.
Prós
 Liberdade de lançamento
Por não se utilizar dos meios oferecidos pelas empresas detentoras dos soft market's, os sistemas web podem ser lançados em menor tempo pois não precisam passar por bateria de exames pelas empresas donas das plataformas para somente após aprovação poderem ser liberados para o público.
EXPERIÊNCIA
Contras
 Uso Off-Line
Apps web não permitem a utilização de funções básicas quando não existe conexão para transmissão de dados. Existem soluções para este problema, mas voltamos ao ponto em que se deixa de ter uma aplicação 100% web e passamos a ter uma aplicação híbrida, que demanda o dobro de custos, manutenção e outros..
Uso por parte do usuário
Uma aplicação 100% web implica no uso do browser nativo da plataforma alvo.
Segundo pesquisa das empresas de análise e market mobile Flurry/comScore NetMarketShare, somente 20% do tempo de um usuário mobile é gasto com o browser e todo o tempo restante é disputado entre jogos (32%), Facebook (18% ) e outras aplicações nativas.
Não se cria atalho para apps web
Aplicações web no mundo mobile não tem o poder de cativar o usuário por não se integrarem perfeitamente ao novo paradigma. Um dispositivo mobile é definido pelas aplicações instaladas nele. Um sistema mobile não cria este vínculo com o usuário, muito menos com seu dispositivo móvel.
PERFORMANCE
 Performance reduzida
Por não utilizar do poder de processamento dos smartphones atuais (alguns dual core), as aplicações web não conseguem usufruir 100% do que os dispositivos móveis têm a oferecer e toda e qualquer forma de criar uma maior integração com o hardware implica no desenvolvimento de duas camadas de programa sendo obviamente uma camada web e uma ponte nativa para ligar a aplicação web ao equipamento físico.
PROFISSIONAIS
Prós
 Maior diversidade de profissionais
A web, por ser uma plataforma com mais de 20 anos de existência possui também uma maior disponibilidade de profissionais altamente especializados, o que auxilia numa futura mudança de equipe sem transtornos,. Este ponto inclusive influencia no item de custo, pois com mais profissionais no mercado a tendência é se ter mais oferta que procura.
ROI
Contras
 Monetização prejudicada
Por não ser distribuída pelos meios “oficiais” uma aplicação web deve “aprender a viver sozinha”. Será necessário desenvolver sua própria forma de monetização da mesma, como apresentação de banner de parceiros ou mesmo para apresentação de campanhas internas, o que na realidade cria mais uma aplicação a ser desenvolvida, mantida e custeada.
SEGURANÇA
 Visibilidade do código É de conhecimento comum a facilidade de se realizar inspeções no código HTML gerado pelo aplicativo, mesmo que não se encontrem falhas de segurança, ainda assim é possível obter informações valiosas.

Inúmeras são as vulnerabilidades existentes no ambiente web. O mesmo se reflete também no mundo mobile, dentre elas as comprovadamente existentes no mundo app web são: Session
Hijacking (roubo de sessão onde é feita a captura de dados de sessão que validem a autenticação do usuário no sistema web, mas que pode ser injetado numa nova sessão para que passe a se “parecer” com a sessão original, dando assim plenos poderes ao usuário mal intencionado); Crosssite request forgery (injeção de requisição em uma requisição corrente, fazendo parecer com que a segunda requisição também seja executada pelo usuário correto); XSS-Cross-site scripting (este ataque permite a injeção de código executável para obter dados sensíveis como senhas e cookies, normalmente usando javascript); Buffer overflow (no caso de aplicações web que consumam outros executáveis nativos que sejam vulneráveis a este tipo de ataque).
VISUAL
 Distorção em dispositivos com configurações de tela e browser diferentes
Mesmo com a utilização de css responsivo, determinadas situações podem gerar telas que se destacam ao rotacionar o dispositivo, ou mesmo em dispositivos cujo tamanho da tela sejam diferentes, ou mesmo que possuam uma implementação diferente do motor web (parte de software responsável por renderizar markups html), em algumas circunstâncias se torna mais trabalhoso que uma tela em código nativo, além de não poder trazer o comportamento esperado. Além de que a busca pela perfeição pela semelhança com a plataforma original pode gerar um efeito conhecido como “Vale da Estranheza”, termo cunhado na robótica que define o ponto limite em que uma semelhança gera mais desconforto que encanto.

Neste meio tempo, inúmeras foram as soluções que surgiram prometendo o WORA (Write Once Run Anywhere) que foi divulgado pela extinta Sun com o seu produto Java ©, porém toda universalização vem com um custo. Em se tratando de dispositivos móveis, temos a perda de performance seguida da grande exposição dos dados sensíveis do sistema. É válido enfatizar o resultado importantíssimo apurado no evento Q3 2012 Developer Survey, a respeito da utilização de aplicativos HTML5:
As gigantes do mundo da informática também tiveram sua cota de testes com o uso de HTML5, para após inúmeras reclamações e problemas decorridos de desempenho abaixo do esperado e falhas de segurança abandonarem seus projetos e recriarem suas aplicações utilizando as tecnologias nativas de cada plataforma.
A Google ©, em detrimento de decidir utilizar a plataforma nativa para o iOS quando do desenvolvimento do Google Maps for iOS desenvolveu um projeto para uma ferramenta que porta código Java (Android) para Objective-C (iOS) e vice-e-versa em detrimento de usar tecnologias que prometem universalidade.
O LinkedIn abandonou sua versão HTML5 do seu cliente mobile e passou a desenvolver as versões nativas do seu sistema. O Facebook também passou por algo semelhante, onde abandonaram projetos HTML5 para criar versões para Android e para iOS separadamente.
"I think the biggest mistake that we made, as a company, is betting too
much on HTML5 as opposed to native, because it just wasn't there…"
Mark Zuckerberg, September 2012
Aplicativos mobile seguros e performáticos não utilizam somente uma tecnologia ou paradigma. Mais sim se apóia em tudo que o mercado oferece que possa trazer benefícios para a sua execução.
Sistemas iOS bem elaborados utilizam a plataforma de desenvolvimento nativo e sistemas Android utilizam o SDK juntamente com o NDK (Native Development Kit), onde partes críticas do código devem ser implementadas em C++, a camada de visão fica por responsabilidade do código Java, a persistência de dados em ambas as plataformas devem se resumir a dados de pouco interesse, e também em ambas os aplicativos são meros clientes consumindo serviços expostos por servidores através de web service Restful (pois consome menos banda em detrimento ao SOAP e que aliado ao uso de TLS e outras estratégias, é tão seguro quanto), o uso de certificação digital fica como um plus, e inevitavelmente técnicas de ofuscamento de código também devem ser implementadas.
